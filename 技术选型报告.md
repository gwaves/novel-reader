## 1. 选型方法论

### 1.1 核心约束
基于 PRD 中的非功能需求 **NFR-DEP-01**（对外仅依赖 LLM API）和 **NFR-DEP-02**（Embedding 本地完成），所有技术选型遵循以下原则：
1. **零外部 SaaS 依赖**：不依赖 Pinecone、Neo4j Aura、云端数据库等外部服务。
2. **嵌入式优先**：数据库、向量索引、模型推理均要求本地内嵌，拒绝 C/S 架构的中间件。
3. **macOS 原生体验**：优先选择能打包为单机 `.app`、启动快、体积可控的方案。
4. **AI 生态成熟度**：LLM 调用、Prompt 工程、文本解析等环节优先选择生态完善的语言栈。

### 1.2 候选技术栈评估维度
| 维度 | 权重 | 说明 |
|---|---|---|
| 本地嵌入能力 | 高 | 能否以库/运行时形式打包进应用 |
| macOS 打包成熟度 | 高 | 能否构建为签名 `.app`，跨芯片架构支持 |
| AI/LLM 生态 | 高 | RAG、Embedding、Prompt 工程工具链丰富度 |
| 图数据库能力 | 高 | Cypher/GQL 支持、性能、可视化 |
| 开发效率 | 中 | 框架上手难度、调试工具、社区活跃度 |
| 包体控制 | 中 | 最终 `.app` 体积是否可控 |

## 2. 技术选型详情

### 2.1 UI 与桌面应用框架

#### 候选方案对比

| 方案 | 技术组合 | 本地嵌入能力 | macOS 体验 | AI 生态 | 包体控制 | 综合评估 |
|---|---|---|---|---|---|---|
| **A. Tauri** | Rust + Web前端(React/Vue) + TypeScript | ⭐⭐⭐ 通过 Rust FFI/侧载 | ⭐⭐⭐ 原生WebView，包体<10MB | ⭐⭐ 前端生态强，但AI后端弱 | ⭐⭐⭐⭐⭐ 极小 | ⚠️ AI后端开发成本高 |
| **B. SwiftUI** | Swift + SwiftData/GRDB | ⭐⭐ 无原生AI/图数据库生态 | ⭐⭐⭐⭐⭐ 最原生 | ⭐ 极弱，需桥接Python | ⭐⭐⭐ 中等 | ⚠️ 功能实现极困难 |
| **C. Electron + Python** | Electron + Python后端 | ⭐⭐⭐ Python生态最强 | ⭐⭐ 非原生，资源占用高 | ⭐⭐⭐⭐⭐ 最强 | ⭐⭐ 极大(300MB+) | ⚠️ 包体过大 |
| **D. PyQt/PySide6** | Python + Qt | ⭐⭐⭐ 完美 | ⭐⭐ 跨平台UI较老 | ⭐⭐⭐⭐⭐ 最强 | ⭐⭐ 大(200MB+) | ⚠️ UI现代感不足 |
| **E. Flutter Desktop** | Dart + FFI | ⭐⭐ FFI可行但复杂 | ⭐⭐⭐ 较流畅 | ⭐⭐ 生态薄弱 | ⭐⭐⭐ 中等 | ⚠️ 图数据库支持差 |

#### 结论与推荐

**推荐方案 C（Electron + Python 后端）作为备选，推荐方案 A（Tauri）作为首选，但存在一个更优的折中方案：Tauri + Python Sidecar。**

经过深入评估，我们提出 **最终推荐方案：Tauri + Python Sidecar 架构**：

- **Tauri** 作为 UI Shell：提供原生 macOS 菜单栏、窗口管理、系统托盘、自动更新、Keychain 访问。前端使用 React + TypeScript，通过 Tauri 的 IPC 与后端通信。
- **Python Sidecar** 作为 AI 引擎：以子进程/嵌入式 Python 运行时形式运行，负责：
  - 文档解析（`ebooklib`, `PyMuPDF`, `beautifulsoup4`）
  - LLM 调用与 Prompt 编排（`openai` SDK + 自定义 Prompt）
  - 实体关系抽取逻辑
  - Embedding 模型推理（`sentence-transformers`）
- **Rust 原生层**：负责图数据库（KuzuDB Rust binding）、向量数据库（LanceDB Rust binding）、本地文件 I/O、配置管理。

**此方案的优势**：
1. Tauri 包体极小（前端 + Rust 二进制通常 < 30MB），Python sidecar 虽大但可剥离为可选下载组件。
2. 前端可视化生态成熟（Cytoscape.js, D3.js, ECharts）。
3. Python AI 生态完整，无需重复造轮子。
4. 通过 Tauri 的 `sidecar` 机制，Python 进程与 Rust 主进程通过 STDIO/WebSocket 通信，架构清晰。

### 2.2 图数据库

#### 核心需求
- 嵌入式：无需独立进程/服务，以库形式链接到应用。
- Cypher 查询：支持 `MATCH (p:Person)-[:MASTER_OF]->(d:Disciple)` 类查询。
- 性能：500+ 节点/2000+ 边的图谱，毫秒级查询。
- macOS 支持：Apple Silicon + Intel 双架构预编译二进制。

#### 候选方案对比

| 方案 | 类型 | Cypher支持 | 嵌入式 | macOS支持 | 性能 | 绑定语言 | 评估 |
|---|---|---|---|---|---|---|---|
| **Neo4j Community** | 原生图数据库 | ⭐⭐⭐ 完整 | ❌ 需JVM进程 | ⭐⭐ 需JRE | ⭐⭐⭐ 强 | Java | ❌ 太重，需JVM |
| **KuzuDB** | 嵌入式图DB | ⭐⭐⭐ 完整 | ⭐⭐⭐ 是，C++核心 | ⭐⭐⭐ 完美 | ⭐⭐⭐ 优秀 | C++/Rust/Python/Node | ✅ **推荐** |
| **DuckDB + 图扩展** | OLAP + 扩展 | ⭐⭐ SQL模拟 | ⭐⭐⭐ 是 | ⭐⭐⭐ 完美 | ⭐⭐⭐ 强 | 多语言 | ⚠️ 无原生图模型 |
| **SQLite + 自制图层** | RDBMS | ❌ 需模拟 | ⭐⭐⭐ 是 | ⭐⭐⭐ 完美 | ⭐⭐ 中小规模可 | 全平台 | ⚠️ 关系查询复杂 |
| **NetworkX** | 内存图库 | ❌ Python API | ⭐⭐⭐ 纯Python | ⭐⭐⭐ 完美 | ⭐⭐ 仅内存 | Python | ⚠️ 无持久化查询语言 |

#### 结论与推荐

**推荐：KuzuDB**

选型理由：
1. **纯嵌入式**：无需守护进程，直接以 Rust crate / Python wheel 形式嵌入应用，启动即可用。
2. **Cypher 完整支持**：支持 `MATCH`, `CREATE`, `MERGE`, `WHERE` 等标准语法，开发体验接近 Neo4j。
3. **存储格式**：单文件 `.kuzu` 数据库，便于备份和迁移。
4. **性能验证**：官方基准显示在百万节点规模下查询 latency < 10ms，满足 NFR-PERF-02。
5. **语言绑定**：Rust binding 成熟（与 Tauri 完美集成），同时有 Python binding（Sidecar 也可用）。

示例 Cypher 查询（人物派系查询）：
```cypher
MATCH (p:Person)-[:BELONGS_TO]->(f:Faction)
WHERE f.name = "青云门"
RETURN p.name, p.aliases, p.description
```

### 2.3 向量数据库与 RAG

#### 核心需求
- 嵌入式：与图数据库同构，本地单文件/目录存储。
- 混合检索：支持向量相似度 + 全文关键词（BM25）联合排序。
- 多租户隔离：每本小说独立命名空间。
- 无需 GPU：纯 CPU 检索性能可接受。

#### 候选方案对比

| 方案 | 类型 | 嵌入式 | 混合检索 | macOS | 多租户 | 语言绑定 | 评估 |
|---|---|---|---|---|---|---|---|
| **LanceDB** | 嵌入式向量DB | ⭐⭐⭐ 是 | ⭐⭐ 向量+过滤 | ⭐⭐⭐ 完美 | ⭐⭐ 表隔离 | Rust/Python/Node | ✅ **推荐** |
| **ChromaDB** | 嵌入式向量DB | ⭐⭐ 需Python运行时 | ⭐⭐ 基础过滤 | ⭐⭐ 依赖Python | ⭐⭐ 集合隔离 | Python/JS | ⚠️ 需Python常驻 |
| **SQLite-vec** | SQLite扩展 | ⭐⭐⭐ 是 | ❌ 仅向量 | ⭐⭐⭐ 完美 | ⭐⭐⭐ 表隔离 | 全平台 | ⚠️ 无全文检索 |
| **Qdrant** | 向量DB | ❌ 需服务端 | ⭐⭐⭐ 强 | ⭐⭐ 客户端可用 | ⭐⭐⭐ 完美 | 多语言 | ❌ 需独立进程 |
| **Milvus Lite** | 嵌入式向量DB | ⭐⭐ 需Python | ⭐⭐ 基础 | ⭐⭐ 依赖Python | ⭐⭐ 集合隔离 | Python | ⚠️ 包体较大 |

#### 结论与推荐

**推荐：LanceDB**

选型理由：
1. **列式存储**：基于 Apache Arrow，向量和元数据同库存储，单表可支撑百万级向量。
2. **Rust 原生绑定**：与 Tauri 架构天然契合，无需跨语言开销。
3. **混合查询能力**：支持 `where` 过滤 + 向量搜索，可结合关键词预过滤实现简易混合检索。
4. **文件格式**：`.lance` 目录格式，每本小说对应一个目录，物理隔离清晰。

**RAG 策略设计**：
```
用户Query
    ├──▶ [本地Embedding模型] ──▶ 查询向量
    │                               │
    │                               ▼
    │                         [LanceDB] 向量检索 Top-20
    │                               │
    ▼                               ▼
[关键词提取] ──▶ [BM25过滤] ──▶ [重排序 Rerank] ──▶ Top-5 上下文
                                                  │
                                                  ▼
                                            [LLM Prompt]
                                                  │
                                                  ▼
                                            [生成答案]
```

### 2.4 本地 Embedding 方案

#### 核心需求
- **完全本地**：模型文件下载到本地，推理在 CPU/NE（Apple Neural Engine）上运行。
- **中文优化**：在中文语义相似度任务上表现良好。
- **体积可控**：模型文件 < 500MB，加载后内存 < 1GB。
- **推理速度**：单句 embedding < 100ms（Apple Silicon）。

#### 候选模型对比

| 模型 | 维度 | 大小 | 中文效果 | 推理框架 | Apple Silicon | 评估 |
|---|---|---|---|---|---|---|
| **BGE-M3 (small/FP16)** | 1024 | ~500MB | ⭐⭐⭐ SOTA | ONNX / sentence-transformers | ⭐⭐ 通过ONNX | ✅ **推荐** |
| **bge-large-zh-v1.5** | 1024 | ~1.3GB | ⭐⭐⭐ 强 | ONNX / sentence-transformers | ⭐⭐ 通过ONNX | ⚠️ 体积偏大 |
| **all-MiniLM-L6-v2** | 384 | ~80MB | ⭐⭐ 一般 | ONNX Runtime | ⭐⭐⭐ 极快 | ⚠️ 中文弱 |
| **multilingual-e5-large** | 1024 | ~2GB | ⭐⭐⭐ 强 | ONNX | ⭐⭐ 较慢 | ⚠️ 体积大 |
| **MLX Embedding (Apple)** |  varies | varies | ⭐⭐⭐ 强 | Apple MLX | ⭐⭐⭐ ANE加速 | ⚠️ 仅Apple Silicon |

#### 推理引擎选型

| 引擎 | 跨平台 | Apple加速 | Python集成 | 包体影响 | 评估 |
|---|---|---|---|---|---|
| **ONNX Runtime** | ⭐⭐⭐ | ⭐⭐ CoreML EP | ⭐⭐⭐ `optimum[onnxruntime]` | ⭐⭐ 中等 | ✅ **推荐** |
| **sentence-transformers** | ⭐⭐⭐ | ❌ PyTorch CPU | ⭐⭐⭐ 完美 | ⭐⭐ 依赖PyTorch大 | ⚠️ 内存占用高 |
| **Apple MLX** | ❌ 仅Apple | ⭐⭐⭐ ANE/GPU | ⭐⭐ `mlx-lm` | ⭐⭐⭐ 极小 | ⚠️ 非跨平台 |
| **llama.cpp (embedding)** | ⭐⭐⭐ | ⭐⭐ Metal | ⭐⭐ 通过绑定 | ⭐⭐⭐ 小 | ⚠️ 集成复杂 |

#### 结论与推荐

**推荐：BGE-M3-ONNX (量化版) + ONNX Runtime**

选型理由：
1. **BGE-M3**：智源开源，中文语义检索效果 SOTA 级别，支持稀疏向量（lexical weight）和密集向量，天然适合做混合检索。
2. **ONNX Runtime**：通过 CoreML Execution Provider 在 Apple Silicon 上获得 Neural Engine 加速，推理速度比纯 PyTorch CPU 快 3-5 倍。
3. **量化模型**：使用 `Optimum` 工具链导出的 ONNX INT8 量化模型，体积可压缩至 200MB 以内，精度损失 < 2%。
4. **Python Sidecar 集成**：在 Python 进程中通过 `optimum[onnxruntime]` 加载模型，Tauri 通过 IPC 调用 embedding 接口。

**备选**：如果确定仅支持 Apple Silicon，可使用 **MLX 生态**（`mlx-embedding`），体积更小、速度更快，但失去 Intel Mac 兼容性。

### 2.5 文档解析引擎

#### 核心需求
- txt/epub/pdf 三种格式解析
- 保留章节层级结构
- 中文编码自动识别（GBK/UTF-8）
- 纯文本提取，去除广告/页眉页脚

#### 候选方案

| 格式 | 推荐库/方案 | 说明 |
|---|---|---|
| **TXT** | Python `chardet` + `pathlib` | 自动检测编码，按空行/章节标题正则切分 |
| **EPUB** | Python `ebooklib` + `beautifulsoup4` | 提取 spine 阅读顺序，按 HTML 结构识别章节标题 |
| **PDF** | Python `PyMuPDF` (fitz) | 提取文字版PDF文本，保留页面布局；扫描版暂不支持（未来可接 `paddleocr` 本地OCR） |

### 2.6 LLM 调用与 Prompt 工程框架

#### 核心需求
- 仅依赖标准 OpenAI-compatible HTTP API
- 支持流式输出（SSE）
- 解析（实体抽取）和对话（问答）共用一套接口，可配置不同模型

#### 选型

**无需重型框架**（如 LangChain/LlamaIndex），原因：
1. 功能边界清晰，不需要链式编排。
2. 减少依赖，降低包体。
3. 原生 `openai` Python SDK（< 1MB）已足够。

**自研轻量层**：
- `LLMClient`：封装 `base_url`, `model_name`, `api_key`，统一处理流式/非流式请求。
- `PromptRegistry`：YAML/JSON 管理 Prompt 模板，支持版本化。
- `Parser`：LLM 输出格式强制为 JSON（JSON Mode / Structured Output），通过 `pydantic` 校验。

示例解析 Prompt 结构：
```yaml
entity_extraction:
  system: "你是一个小说内容解析专家。请从以下文本中提取实体，返回JSON。"
  schema:
    persons: [{name, aliases:[], faction, description}]
    factions: [{name, description}]
    items: [{name, type, owner, description}]
    skills: [{name, type, practitioner, description}]
```

### 3.2 数据流设计

**导入阶段**：
```
EPUB/TXT/PDF
    └──▶ Python Sidecar (解析)
            ├──▶ 章节树 (JSON) ──▶ Rust ──▶ SQLite/JSON (章节元数据)
            ├──▶ 文本段落 ──▶ ONNX Embedding ──▶ 向量 ──▶ LanceDB
            └──▶ 全文文本 ──▶ LLM 实体抽取 ──▶ 实体/关系 JSON
                        │
                        ▼
                    Rust ──▶ KuzuDB (图谱写入)
```

**查询阶段**：
```
用户输入
    ├──▶ 图谱查询: "青云门有哪些人？"
    │       └──▶ Rust ──▶ KuzuDB (Cypher) ──▶ 图谱可视化
    │
    ├──▶ RAG 问答: "九转玄功是谁的？"
    │       ├──▶ Python: Query Embedding (ONNX)
    │       ├──▶ Rust: LanceDB 向量检索 Top-20
    │       ├──▶ Python: BM25 重排序 Top-5
    │       └──▶ Python: LLM 生成答案 (带原文引用)
    │
    └──▶ 混合查询: "主角和青云门的关系？"
            ├──▶ KuzuDB 关系查询
            ├──▶ LanceDB 相关段落检索
            └──▶ LLM 综合生成
```

### 3.3 部署与打包方案

| 组件 | 打包方式 | 体积预估 |

## 4. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| Python Sidecar 跨进程通信延迟 | 中 | 使用共享内存/Unix Socket 替代 STDIO；批量请求合并 |
| Apple Intel Mac ONNX 性能差 | 中 | 提供纯 PyTorch fallback；推荐用户使用 Apple Silicon |
| LLM 解析大文本 Token 超限 | 高 | 文本预分块（< 4K tokens/块），分块多次调用后合并 |
| KuzuDB 大规模写入性能瓶颈 | 低 | 批量写入（Transaction），解析时先写 JSON 缓存后批量导入 |
| EPUB 格式混乱导致解析失败 | 中 | 提供手动章节编辑界面，允许用户调整切分点 |

## 5. 选型结论

| 模块 | 推荐技术 | 备选方案 |
|---|---|---|
| **UI 框架** | Tauri + React + TypeScript | Electron + React |
| **图数据库** | KuzuDB (嵌入式, Cypher) | Neo4j Desktop (非嵌入式) |
| **向量数据库** | LanceDB (嵌入式) | ChromaDB (需Python常驻) |
| **本地 Embedding** | BGE-M3 ONNX INT8 + ONNX Runtime | MLX Embedding (仅Apple Silicon) |
| **文档解析** | Python `ebooklib` + `PyMuPDF` | 自研 Rust 解析（长期） |
| **LLM 调用** | `openai` SDK + 自研 Prompt 层 | LiteLLM (多提供商适配) |
| **AI 运行时** | Python Sidecar (PyInstaller) | Rust 原生 ONNX (长期演进) |
| **图谱可视化** | Cytoscape.js (Web) | D3.js / ECharts Graph |

**整体架构风格**：Tauri 作为轻量级原生壳，Python Sidecar 承载 AI 能力，Rust 核心层管理数据持久化，形成 **"轻壳 + 重引擎 + 强数据"** 的分层架构。
