## 1. 设计目标与约束

### 1.1 设计目标
- **分层清晰**：UI 层、应用服务层、AI 引擎层职责分离，单向依赖。
- **数据一致性**：图谱、向量、章节元数据三者在同一本小说内保持事务一致性。
- **扩展性**：未来支持新的小说格式、新的可视化类型、新的 Embedding 模型时，改动范围可控。
- **可测试性**：各层通过接口契约交互，支持单元测试和集成测试分离。

### 1.2 设计约束
- 对外仅暴露 LLM HTTP API，所有状态本地持久化。
- Python Sidecar 作为独立进程，可能崩溃/重启，Rust 主进程需具备自愈能力。
- macOS 沙盒与 Keychain 访问规则限制。
- 单用户单机，无需考虑并发写冲突（但需防范异常关闭导致的数据损坏）。

### 2.2 模块职责

| 模块 | 语言 | 职责 | 关键设计决策 |
|---|---|---|---|
| **Presentation** | TypeScript/React | 所有用户界面、状态管理、可视化渲染 | 使用 Zustand 管理全局状态，Cytoscape.js 负责图渲染 |
| **Tauri Commands** | Rust | 暴露安全的前端可调用接口，权限校验 | 每个 Command 都校验 `novel_id` 合法性 |
| **NovelService** | Rust | 小说导入、删除、章节树管理、解析任务调度 | 使用 tokio 异步任务队列，避免阻塞 UI |
| **GraphService** | Rust | 实体 CRUD、关系 CRUD、Cypher 查询执行 | 直接调用 KuzuDB Rust API，封装事务 |
| **RagService** | Rust | 向量检索、混合检索、结果组装 | 调用 LanceDB Rust API，关键词过滤通过 KuzuDB 元数据辅助 |
| **ConfigManager** | Rust | 配置读写、LLM 配置加密存储 | API Key 仅存 Keychain，其余配置存 SQLite/JSON |
| **SidecarManager** | Rust | Python 进程生命周期管理、健康检查、请求路由 | 进程崩溃自动重启，请求排队等待恢复 |
| **ParseEngine** | Python | EPUB/TXT/PDF 解析、章节结构化 | 输出标准化 `NovelDocument` JSON |
| **EmbedEngine** | Python | 文本分块、ONNX Embedding 推理 | 批量推理优化吞吐量，支持流式回调进度 |
| **LLMEngine** | Python | Prompt 组装、API 调用、流式/非流式响应 | 解析用 JSON Mode，对话用 SSE 流式 |
| **RankEngine** | Python | BM25 重排序、交叉编码器重排序 | 纯 CPU 实现，轻量快速 |

### 2.3 跨进程通信机制

Rust 主进程与 Python Sidecar 之间的通信采用 **Unix Domain Socket (UDS) + JSON-RPC 2.0**。

**选型理由**：
- UDS 比 STDIO 更稳定，支持双向流式通信（SSE 透传）。
- UDS 比 TCP localhost 更安全（文件权限控制，不暴露端口）。
- JSON-RPC 2.0 轻量、无IDL编译，适合快速迭代。

**通信协议栈**：
```
┌─────────────────────────────────────────┐
│  Application Payload (JSON-RPC Request) │
├─────────────────────────────────────────┤
│  JSON-RPC 2.0 Framing                   │
├─────────────────────────────────────────┤
│  UDS Stream (Unix Domain Socket)        │
├─────────────────────────────────────────┤
│  Unix File System (Abstract Namespace)  │
└─────────────────────────────────────────┘
```

**进程生命周期状态机**：
```
          ┌─────────────┐
          │   Stopped   │
          └──────┬──────┘
                 │ start_sidecar()
                 ▼
          ┌─────────────┐
          │  Starting   │
          └──────┬──────┘
                 │ health_check OK
                 ▼
          ┌─────────────┐     health_fail x3
          │   Ready     │ ─────────────────▶ Restarting
          └──────┬──────┘
                 │ stop_sidecar()
                 ▼
          ┌─────────────┐
          │   Stopped   │
          └─────────────┘
```

## 3. 主要数据结构

### 3.1 核心实体定义

所有跨层传输的数据结构使用 JSON 序列化，以下为 TypeScript Interface 定义（作为跨语言契约的 Source of Truth）。

#### Novel (小说)

```typescript
interface Novel {
  /** 全局唯一ID，UUID v4 */
  id: string;
  /** 小说标题 */
  title: string;
  /** 作者（如可提取） */
  author?: string;
  /** 原始文件路径 */
  sourcePath: string;
  /** 文件格式 */
  format: 'txt' | 'epub' | 'pdf';
  /** 总字数 */
  totalChars: number;
  /** 总章节数 */
  totalChapters: number;
  /** 解析状态 */
  status: NovelStatus;
  /** 创建时间 */
  createdAt: number; // unix timestamp (ms)
  /** 最后更新时间 */
  updatedAt: number;
  /** 各阶段进度 */
  progress: ParseProgress;
}

type NovelStatus = 'imported' | 'parsing' | 'embedding' | 'extracting' | 'completed' | 'error';

interface ParseProgress {
  chaptersExtracted: number;   // 章节提取 0-100
  vectorsIndexed: number;      // 向量索引 0-100
  entitiesExtracted: number;   // 实体抽取 0-100
}
```

interface Chapter {

interface Entity {

#### 常用查询模板

```cypher
// 1. 查询某派系所有成员
MATCH (p:Person)-[r:PersonFaction]->(f:Faction)
WHERE f.name = '青云门' AND r.type = 'belongs_to'
RETURN p.name, p.aliases, p.description

// 2. 查询某人物的所有关系网络（一度关系）
MATCH (p:Person {name: '李逍遥'})-[r:PersonPerson]-(target:Person)
RETURN target.name, r.type, r.description

// 3. 查询某人掌握的所有功法
MATCH (p:Person)-[r:PersonSkill]->(s:Skill)
WHERE p.name = '李逍遥'
RETURN s.name, s.description, r.type

// 4. 查询某道具的历任主人
MATCH (p:Person)-[r:PersonItem]->(i:Item)
WHERE i.name = '诛仙剑'
RETURN p.name, r.type, r.chapterIndex
ORDER BY r.chapterIndex

// 5. 派系势力统计
MATCH (f:Faction)<-[r:PersonFaction]-(p:Person)
WHERE r.type IN ['belongs_to', 'leader_of']
RETURN f.name, COUNT(p) AS memberCount
ORDER BY memberCount DESC

// 6. 人物图谱可视化数据导出（含节点坐标由前端计算）
MATCH (p:Person)
RETURN p.id, p.name, p.aliases, 'Person' AS label
UNION ALL
MATCH (f:Faction)
RETURN f.id, f.name, f.aliases, 'Faction' AS label;

MATCH (a)-[r]->(b)
RETURN a.id, b.id, r.type, r.description
```

### 4.2 LanceDB 向量存储 Schema

每本小说对应独立的 LanceDB 目录，路径：
```
~/Library/Application Support/NovelReader/novels/{novelId}/vectors.lance/
```

#### 表结构

```python
import lancedb
import pyarrow as pa

schema = pa.schema([
    pa.field("id", pa.string()),              # 段落全局ID
    pa.field("novelId", pa.string()),
    pa.field("chapterIndex", pa.int32()),
    pa.field("paragraphIndex", pa.int32()),   # 段落全局序号
    pa.field("text", pa.string()),            # 原文内容（加密存储或引用）
    pa.field("vector", pa.list_(pa.float32(), 1024)),  # BGE-M3 向量
    pa.field("charCount", pa.int32()),
    pa.field("createdAt", pa.int64()),
])

# 创建表时建立向量索引
table = db.create_table("paragraphs", schema=schema)
table.create_index(
    metric="cosine",
    vector_column_name="vector",
    index_type="ivf_pq",   # IVF + Product Quantization，适合百万级
    num_partitions=64,     # 根据数据量调整
    num_sub_vectors=16
)
```

#### 检索接口

```python
# 语义检索
results = table.search(query_vector) \
    .where(f"novelId = '{novelId}'") \
    .limit(20) \
    .to_list()

# 混合检索（先向量召回，再用关键词过滤）
results = table.search(query_vector) \
    .where(f"novelId = '{novelId}' AND text LIKE '%{keyword}%'") \
    .limit(50) \
    .to_list()
```

## 7. 错误码与状态定义

### 7.1 全局错误码

| 错误码 | 说明 | HTTP类比 |
|---|---|---|
| `OK` | 成功 | 200 |
| `INVALID_PARAMS` | 参数缺失或格式错误 | 400 |
| `NOT_FOUND` | 资源不存在（小说/实体/会话） | 404 |
| `ALREADY_EXISTS` | 资源已存在 | 409 |
| `SIDECAR_UNAVAILABLE` | Python Sidecar 未就绪 | 503 |
| `SIDECAR_TIMEOUT` | Sidecar 请求超时 | 504 |
| `LLM_ERROR` | LLM API 调用失败（含网络/认证/限流） | 502 |
| `PARSE_ERROR` | 文档解析失败 | 422 |
| `DB_ERROR` | 数据库操作失败 | 500 |
| `IO_ERROR` | 文件读写失败 | 500 |
| `UNKNOWN_ERROR` | 未分类错误 | 500 |

### 7.2 解析状态机

```
           import_novel()
               │
               ▼
    ┌─────────────────┐
    │    imported     │ ◀── 文件已复制，章节结构已提取
    └────────┬────────┘
             │ start_parsing()
             ▼
    ┌─────────────────┐
    │    parsing      │ ◀── LLM 抽取实体/关系（可中断）
    └────────┬────────┘
             │ entities done
             ▼
    ┌─────────────────┐
    │   embedding     │ ◀── 文本向量化（可中断）
    └────────┬────────┘
             │ vectors done
             ▼
    ┌─────────────────┐
    │  extracting     │ ◀── 最终数据聚合入库
    └────────┬────────┘
             │
             ▼
    ┌─────────────────┐
    │   completed     │ ◀── 可正常使用
    └─────────────────┘
             │
    reparse_novel()
             ▼
    ┌─────────────────┐
    │     error       │ ◀── 任何阶段失败进入此状态，保留已完成的阶段
    └─────────────────┘
```

### 7.3 Sidecar 状态定义

| 状态 | 说明 | 可接受请求？ |
|---|---|---|
| `stopped` | 未启动 | 否 |
| `starting` | 启动中 | 否（排队） |
| `ready` | 健康，空闲 | 是 |
| `busy` | 健康，处理中 | 是（排队） |
| `error` | 健康检查失败 | 否 |
| `restarting` | 自动重启中 | 否（排队） |

## 8. 安全设计

### 8.1 数据加密

| 数据类型 | 加密方式 | 密钥管理 |
|---|---|---|
| 小说原文 (chapters/*.enc) | AES-256-GCM | 基于 `novelId` + 设备唯一标识派生密钥 |
| API Key | Keychain (kSecClassGenericPassword) | macOS 系统 Keychain |
| 向量/图谱数据 | 不加密（仅本地索引） | - |
| 配置/元数据 | 不加密 | - |

### 8.2 文件系统权限

- 应用仅请求 `~/Library/Application Support/NovelReader/` 目录的读写权限。
- 用户导入小说时通过 Tauri 的 `dialog.open()` 获取文件路径（macOS 沙盒自动授权单次访问）。
- 不请求全局磁盘访问权限。

### 8.3 LLM 数据传输安全

- API Key 仅存在于 Rust 进程内存和 Keychain 中，**绝不写入日志或持久化文件**。
- Python Sidecar 接收的 `modelConfig.api_key` 仅在请求时通过 UDS 传输（UDS 不经过网络）。
- 支持用户配置自定义 `base_url` 指向本地代理（如 Ollama、LM Studio）。
